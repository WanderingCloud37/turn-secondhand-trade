# 校园二手物品交易系统 — PPT 项目简报

## 一、项目一句话简介

面向高校学生的校园二手物品在线交易平台。学生可注册登录、发布闲置商品、按分类/关键字检索、在线下单交易；管理员可审核商品、管理用户、处理订单、发布公告。

---

## 二、功能模块清单

### 用户认证
- 注册（用户名、密码、手机号，密码 BCrypt 哈希存储）
- 登录（校验密码 + 读取角色权限）
- 忘记密码（邮箱验证码重置）
- 登录页密码框回车触发登录

### 商品浏览与检索
- 分页展示商品列表，每页 10 条，仅显示在售商品
- 按商品名模糊搜索
- 按分类下拉筛选（7 个商品分类，动态加载）
- 商品缩略图展示，鼠标悬停/点击查看详情
- 商品 ID 四位补零显示

### 商品发布与审核
- 卖家发布商品：标题、价格、分类、描述、上传图片（≤3MB）
- 发布后状态为"待审核"，需管理员审核通过才能上架
- 管理员审核——通过（→在售）或拒绝（→已拒绝），记录审核人

### 订单交易
- 买家下单 → 创建订单（待付款）
- 卖家确认付款 → 待收货
- 买家确认收货 → 已完成
- 下单使用数据库事务（UPDLOCK）保证并发安全

### 管理员后台
- 商品审核（列表 + 通过/拒绝按钮）
- 用户管理（搜索 + 禁用/解禁）
- 订单管理（全量订单查看 + 标记完成）
- 数据统计（用户数/商品数/订单数 + 分类统计）
- 公告发布（超级管理员专有）

### 角色权限体系
- 消费用户：浏览、购买
- 商品管理员：仅可审核商品
- 超级管理员：全部管理功能 + 公告发布
- 登录后主界面按钮按角色显隐，管理员面板 Tab 按权限可见

## 四、数据库设计（中文概述）

系统共 8 张业务表：

| 表 | 作用 | 关键关系 |
|----|------|---------|
| `roles`（角色） | 定义三种角色及四项权限标志 | 被 users 引用 |
| `users`（用户） | 账号信息、BCrypt 密码哈希、角色关联 | 外键 → roles |
| `categories`（商品分类） | 电子产品/教材教辅/生活用品等 7 类 | 被 goods 引用 |
| `goods`（商品） | 标题/价格/描述/状态（待审核→在售→已售），关联卖家、分类 | 外键 → users, categories |
| `goods_images`（商品图片） | 支持一商品多图，存储文件路径 | 外键 → goods，级联删除 |
| `orders`（订单） | 买卖双方、成交价格、状态流转、收货地址、时间戳 | 外键 → goods, users |
| `announcements`（公告） | 超级管理员发布的系统公告 | 外键 → users |
| 视图 × 2 | 在售商品视图、订单详情视图 | 用于列表查询 |

**核心外键关系：** 用户→发布→商品→产生→订单←购买←用户，角色→分配→权限控制

## 五、安全与完整性设计

### 密码安全
- 用户密码采用 BCrypt 算法生成单向哈希，明文永不可逆
- 登录时比对哈希值，不存储不传输明文密码

### SQL 注入防护
- 所有数据库操作均使用 SqlParameter 参数化查询
- 无任何 SQL 字符串拼接

### 访问控制
- 登录时读取角色名（JOIN roles 表），存入全局变量
- 主界面管理员按钮按角色显隐
- 管理员面板加载时二次校验，非管理员自动关闭
- 管理员不能禁用自己或超级管理员

### 数据完整性
- 所有表设主键 + 外键约束 + UNIQUE 约束
- goods/orders 之间的 UNIQUE 约束保证一商品不重复下单
- 下单使用事务 + UPDLOCK 锁保证并发不超卖
- 用户/商品采用软删除策略：status 标记替代物理 DELETE

### 错误处理
- DataGridView 默认错误对话框已屏蔽
- 订单失败自动回滚事务
- 图片不存在时显示默认占位图

## 七、答辩可重点展开的方向

### 方向 1：商品审核机制
> 为什么不做直接上架？——防止违规/低质商品。发布→待审核→管理员审核→上架/拒绝，完整闭环。

### 方向 2：分类检索功能
> 从"只能搜标题"到"分类下拉 + 关键字联合过滤"，如何用一条 SQL 的 `@c = 0 OR category_id = @c` 实现零冗余筛选。

### 方向 3：数据库安全防护
> 不用明文存密码（BCrypt 哈希）、不用拼 SQL（SqlParameter 参数化）、不让一个人干所有事（角色权限细分）。三层防线。

### 方向 4：下单的事务处理
> 为什么下单要包在事务里？——防止并发超卖。SELECT + UPDLOCK → 查库存 → INSERT 订单 → UPDATE 商品状态，四步全成功才 COMMIT。
